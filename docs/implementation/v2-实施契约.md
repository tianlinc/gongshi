## API 端点

### `GET /`

- 用途：页面入口：未登录返回 templates/login.html；已登录跳 /dashboard（302）
- 请求：无
- 成功响应：text/html 登录页 或 302 → /dashboard
- 错误响应：无
- 调用的 RDMClient 方法：无

### `GET /dashboard`

- 用途：工时主表页面壳：未登录跳 /；已登录返回 templates/dashboard.html
- 请求：无
- 成功响应：text/html 主表页
- 错误响应：302 → /
- 调用的 RDMClient 方法：无

### `POST /api/login`

- 用途：用户登录 RDM。成功后写 flask.session['username']+'logged_in'，clients[username]=RDMClient 实例。
- 请求：JSON {username: str, password: str}
- 成功响应：{success:true, username:str}
- 错误响应：{success:false, message:'用户名或密码错误' | '用户名或密码不能为空' | '网络异常，请稍后重试'} 状态码 200
- 调用的 RDMClient 方法：RDMClient.login(username, password)

### `POST /api/logout`

- 用途：清除 flask.session 与 clients[username]。注意：localStorage 记住的账号密码由前端管理，本接口不动。
- 请求：无 body
- 成功响应：{success:true}
- 错误响应：无（始终成功）
- 调用的 RDMClient 方法：无

### `GET /api/tasks`

- 用途：拉取当前用户在 RDM 的【我负责】+【我参与】任务，去重合并，过滤 status ∈ {RN, NS}（即 进行中/未启动）。
- 请求：无 query。未登录由 before_request 统一拦截。
- 成功响应：{success:true, tasks:[{task_id:str, name:str, project:str, status:'未启动'|'进行中', source:'responsible'|'participate'|'both'}]}
- 错误响应：未登录 → {success:false, message:'未登录'}（401 或 200，前端按 message 判断）；其他 → {success:false, message:'任务加载失败：xxx'}
- 调用的 RDMClient 方法：RDMClient.get_my_tasks()

### `GET /api/timesheet`

- 用途：按周回填 RDM 中已存在的工时（D3）。week_start 必须为周一 YYYY-MM-DD；后端容错：若收到非周一日期，自动 normalize 到该周周一。
- 请求：query: week_start=YYYY-MM-DD（周一）
- 成功响应：{success:true, week_start:str, tasks:[{task_id:str, hours:[7 个 number, 周一→周日, 未填=0], completion_rates:[7 个 number|null]}]}（无回填记录时 tasks: []）
- 错误响应：未登录 → {success:false, message:'未登录'}；参数缺失 → {success:false, message:'缺少参数 week_start'}；RDM 异常 → {success:false, message:'回填查询失败：xxx'}
- 调用的 RDMClient 方法：RDMClient.get_week_existing(week_start)

### `POST /api/submit-day`

- 用途：逐天逐任务提交单日工时到 RDM（覆盖语义：带 reportId 走更新路径，第一天创建后续自动更新）。前端按任务串行 + 任务内按天串行调用，本端点不重试。
- 请求：JSON {week_start:'YYYY-MM-DD'(周一), task_id:str, name:str, day_index:int(0-6, 周一=0), date:'YYYY-MM-DD', hour:number(>0,<=24,0.5 步进), completion_rate:int|null}
- 成功响应：{success:true, message:'提交成功', task_id:str, day_index:int}
- 错误响应：未登录 → {success:false, message:'未登录'}；字段非法 → {success:false, message:'字段非法：xxx'}；RDM 写入失败 → {success:false, message:'RDM 写入失败：<中文原因>'}（不抛 Traceback）
- 调用的 RDMClient 方法：RDMClient.get_week_existing(week_start) + RDMClient.submit_day(task_id, day_index, date, hour, completion_rate, week_start)

## RDMClient 方法

### `encrypt`

- 签名：`encrypt(self, text: str) -> str`
- 对接的 RDM 端点：无（纯本地：base64 → AES-ECB(key=b'abcdefgabcdefg12') → base64）
- 返回：加密后的 base64 字符串，用于 j_username/j_password
- 风险：已 v1 实现，保持不变。注意 padding 用 pycryptodome 的 pad(...,AES.block_size)。

### `login`

- 签名：`login(self, username: str, password: str) -> dict`
- 对接的 RDM 端点：GET /index.jsp（拿 cookie/JSESSIONID）+ POST /j_security_check（form: j_username, j_password 双重编码；isExpires=1, BROWSER_VERSION=1, REMOTE_LANGUAGE=zh-cn）
- 返回：{'success':bool, 'message':str}
- 风险：v1 已稳定，但需要 v2 新增：① self.session.headers 加浏览器 UA + Accept-Language=zh-CN（架构方案 §2.2）；② 统一 r.encoding = r.apparent_encoding or 'gbk'（§4.4）。错误判定：'error=true' in resp.url 或 '无效用户名或密码' in resp.text 视为账号密码错误；'loginForm' in resp.text 且 len < 15000 视为通用登录失败。

### `_request`

- 签名：`_request(self, method: str, path: str, **kwargs) -> requests.Response`
- 对接的 RDM 端点：内部辅助：所有对 RDM 的 HTTP 请求都过这层
- 返回：requests.Response（已统一 encoding）
- 风险：§4.5 connection-level retry：捕获 ConnectionError/Timeout 时最多重试 2 次，间隔 1s；HTTP 4xx/5xx 不重试。所有响应统一 r.encoding = r.apparent_encoding or 'gbk'。

### `get_my_tasks`

- 签名：`get_my_tasks(self) -> list[dict]`
- 对接的 RDM 端点：① GET /pages/task/list/myTask.jsf（取 javax.faces.ViewState）→ ② POST /pages/task/list/myTask.jsf（A4J，cate=0 我负责）→ ③ POST 同上（cate=22 我参与）。POST headers 必须含 Faces-Request: partial/ajax + X-Requested-With: XMLHttpRequest，content-type application/x-www-form-urlencoded
- 返回：list[dict] 每项含 task_id, name, project, status('未启动'|'进行中'), source('responsible'|'participate'|'both')。已去重合并、已按 status ∈ {RN, NS} 过滤、已映射 RN→进行中、NS→未启动
- 风险：参考 tools/captured/a4j_myTask_refreshBody.xml：响应是 XML 包 HTML，<tr class='body-row' id='{taskId}' status='RN'> 是 task_id+status 的来源。POST 表单字段：operate=operate, operate:refreshBody=operate:refreshBody, javax.faces.ViewState=<上次值>, cate=0|22, type=TSK, module=TSK, refreshType=0, isInit=Y, page=1。其他 hidden 字段不传可能 NPE，照抄 captured xml 的请求示例传空串。状态码白名单 {'RN','NS'}；遇到未知 status 应 logging warning 并默认保留（架构 §9.4）。

### `get_week_existing`

- 签名：`get_week_existing(self, week_start: str) -> dict`
- 对接的 RDM 端点：GET /pages/myspace/log/workLogView.jsf?isView=N&userId=&reportId=&startDate=YYYY-MM-DD（startDate 必须周一，否则被服务端 normalize）
- 返回：{'view_state':str, 'report_id':str|'', 'login_user_id':str, 'data_json':str(原始字符串透传给 saveLog), 'entries':[{task_id:str, hours:[7], completion_rates:[7]}]}（兼顾 submit_day 透传 view_state/report_id 的需要）
- 风险：data 字段在 <input id="data" value="..."> 中，HTML 实体编码，需 html.unescape + json.loads。data.unBody 是已填的计划外任务数组：每行的 rs/es/ms/is 字段形如 [[day, value], ...]，按 day 索引（0-6）填回长度 7 的数组。空 unBody 时 entries=[]。view_state 用 BeautifulSoup 找 input[name='javax.faces.ViewState'] 取 value。fixture 参考 tools/captured/pages__myspace__log__workLogView.jsf。loginUser 来自隐藏字段 input[name='loginUser']。

### `submit_day`

- 签名：`submit_day(self, task_id: str, name: str, day_index: int, date: str, hour: float, completion_rate: int|None, week_start: str) -> dict`
- 对接的 RDM 端点：POST /pages/myspace/log/workLogView.jsf（A4J，operate:saveLog=operate:saveLog；headers 同 myTask 的 A4J POST）
- 返回：{'success':bool, 'message':str}
- 风险：实现路径：① 调用 get_week_existing(week_start) 获取最新 view_state + report_id + data_json + login_user_id（每次提交都重新拿，因 ViewState 不可缓存）；② 用 rdm/worklog.encode_unplanned_info() 拼出 _unplannedInfo 字符串——本次提交只含 1 个 TaskEntry 且 day-csv 只含 day_index 一个数字，_rate{day}/_effort{day}/_remark{day}/_issue{day} 仅一组；③ 拼 saveLog 表单：operate=operate, operate:saveLog=operate:saveLog, javax.faces.ViewState=<最新>, startDate=week_start, isView=N, userId=, isOwner=Y, objectId=<report_id 或 ''>, summary='', workRemark='<7 段空格用 #-%@-#@ 连接>', issueRemark='<同>', rightUserId='', unplannedInfo=<上述字符串>, loginUser=<login_user_id>, data=<透传 data_json>；④ POST，状态码 200 + 响应不含 'viewExpired' / 'loginForm' 视为成功。失败原因翻译：viewExpired→'会话已过期，请刷新重试'；loginForm→'未登录'（前端按 401 处理）；HTTP 5xx→'RDM 服务异常，请稍后重试'。__delimiter__: S1='#-@%!-@#', S2='#-%@-#@', S3='#-%#!#-@-#@', S4='#-%#!A-#-@-#@'；序列化模板：{task_id}{S1}{day_index}{S1}{day_index}{S2}_rate{S4}{rate or ''}{S2}_effort{S4}{hour}{S2}_remark{S4}{S2}_issue{S4}（架构 §2.5）。RISK：本期不实现 object_id/type_id 字段（架构 §9.5 删除分支不开），首次提交时若 RDM 强制要求 object_id（项目归属），需要从 get_my_tasks 的 tr td 第 8 列 openEntity('projectId','PJT') 中再抓出 project_id 一并带进去——验收时才能确认是否必填。

## 关键决策

1. **Session 模型** — 沿用 v1 模块级 clients dict，flask.session 仅存 username + logged_in。Flask 重启 → clients 清空 → 用户在 /api/* 收到 {success:false, message:'未登录'} → 前端跳 /。不引入 ADR-0001 决策 A 的凭据回放（需求 §5 明确 session 不重构）。
   理由：CLAUDE.md 'session trap' 段落明示这是已知行为；需求 v2.2 §5 列为不变契约。

2. **未登录响应状态码** — 统一返回 HTTP 200 + JSON {success:false, message:'未登录'}（不用 401）。前端根据 message 字段跳转，与 prototype/assets/app.js 中 apiCall 的判断一致。
   理由：prototype 第 155 行 if (data && data.success === false && data.message === '未登录') 已是这种约定；后端跟从前端契约更省事，避免改动 fetch 错误处理逻辑。

3. **记住账号密码** — 纯前端 localStorage 明文存储，key: gongshi_remember_username / gongshi_remember_password / gongshi_remember_flag。后端不参与、不持久化。登录页明确告知用户'密码以明文形式保存在本浏览器'。提供'清除已保存账号'按钮。
   理由：需求 A6 + 设计 §3.1.1。安全风险由用户接受（个人提效工具，非多人共享）。

4. **提交策略** — 前端按'任务 → 任务内按天'双层串行：每天调一次 /api/submit-day，遇任意一天失败立即停止，不继续提交该任务剩余天与后续任务。后端不重试、不批量。
   理由：prototype/assets/app.js 第 127 行已是 /api/submit-day 单天端点；用户最新决定（覆盖了架构 §9.3 的'每任务内重试 2 次'）：任何失败立即停。逐天提交简化了覆盖语义——RDM 拿到 reportId 后续天自动覆盖。

5. **_unplannedInfo 编码位置** — 放在 app.py 内作为模块级常量 + 内部函数 encode_unplanned_info(...)，不拆出 rdm/worklog.py 子模块。
   理由：CLAUDE.md 强调 'Single-file backend'；架构 §3.1 拆 5 模块的方案被需求 §5 'session 不重构' 隐式收回。预计 app.py 总行数 < 600，不触发拆分阈值。

6. **完成率字段形状** — 前端 entries[task_id].rates 长度固定 7，元素 int|null（小时为 0 那天 rate 也保留可填，但提交时若 hour<=0 跳过当天）。后端 /api/submit-day 单天接口的 completion_rate 字段为 int|null。
   理由：需求 C2 + 设计决策 D-01；与 prototype state.entries 完全一致。

7. **GBK 编码 + 控制台输出** — app.py 内所有 RDM 响应 r.encoding = r.apparent_encoding or 'gbk'；所有 print/logging 用 [OK]/[X]，禁用 ✓✗ 等 unicode 箭头；UI 字符串中文。
   理由：CLAUDE.md 'Windows console can't print Unicode arrows' + 架构 §4.4。

8. **日期 normalize** — 前端用 ISO 周（getWeekMonday）算出周一传给后端；后端 /api/timesheet 与 /api/submit-day 收到 week_start 后再次保险 normalize：datetime.strptime(week_start,'%Y-%m-%d')，weekday() 取 0 表示周一，否则减回到周一。
   理由：防御 RDM 服务端的 normalize 行为不一致；与架构 §2.7 一致。

9. **RDM base URL** — 硬编码 'http://10.111.36.3:2029' 在 RDMClient.__init__ 默认值，不引入 config.py。
   理由：CLAUDE.md '不可配置' + 需求 §5。

10. **prototype/ 目录** — 原样保留作为视觉回归基线，不被覆盖、不被引用。所有上线代码在 templates/ + static/ 下。
   理由：设计 §7 + 任何前端调整可直接对照 prototype/ 验证一致性。

## 已知风险

- submit_day 是否必须传 object_id（任务归属项目 ID）和 type_id 才能写入 RDM——架构 §4.3 提示 RDM 部分项目可能强制走计划内分解，本期假设所有 myTask 任务都允许 _unplannedInfo 写入。验收时若某任务写不进，需要从 myTask 的 td 列扩抓 project_id（openEntity('projectId','PJT')）并加入 TaskEntry。
- RDM 任务状态码白名单只验证过 'RN'。'NS'（未启动）、'FN'（已完成）等取值是从 myTask.html 菜单按钮 ID 反推。实跑时遇到 RN/NS 之外的 status，应 logging warning 并默认保留该任务（宁滥勿缺），事后人工对照 RDM 网页校准白名单。
- _unplannedInfo 4 个 delimiter（S1/S2/S3/S4）来自 unplannedTask.js 的逆向产物，未做严格 fuzzing。若 RDM 升级改 delimiter 或字段名，submit_day 整条会瞬间失败。建议把 tools/captured/ 提交到 git 当快照，未来出问题先 diff。
- workRemark / issueRemark 字段一周 7 段用 '#-%@-#@' 分隔，本期默认填 7 个空格段。未验证空段是否触发 RDM 服务端校验失败；若失败需要改填 7 个空字符串或单空格——首次提交时由 dev-engineer 实跑微调。
- 逐天提交意味着同一任务 1 天 1 次 GET workLogView + POST saveLog；7 天满填 = 14 次 HTTP。若用户一周填 6 个任务每天 7 天 = 84 次内网调用。在内网延迟 50ms 下约 4-5s，可接受；但若 ViewState 在第 N 次后失效会出现奇怪 422，需要 dev-engineer 关注 saveLog 响应里 viewExpired 字样。
- 二次提交覆盖语义：第一天 submit 创建 reportId，从第二天起 get_week_existing 应该能拿到该 reportId 自动走更新。但若 RDM 在 reportId 创建后有缓存延迟（unlikely 但有风险），第二天 get_week_existing 仍可能返回 report_id=''——表现为'第二天又新建一条'。验收时需在同一周连填 2 天后到 RDM 网页看是否合并到一条 reportId。
- login_user_id 字段从 workLogView 的 hidden input[name='loginUser'] 取。captured 文件 pages__myspace__log__workLogView.jsf 必须包含此字段；若 RDM 在某些场景下不渲染（如 reportId 为空 + userId 为空时），submit_day 会 KeyError。dev-engineer 实现 get_week_existing 时应 fallback：找不到 loginUser 时返回空字符串，submit 时直接传空——按 unplannedTask.js 逻辑这是被允许的。
- 前端 /api/timesheet 端点在原型中按 mock JSON 索引；真实接口里若用户在 RDM 网页填了部分天但部分天为 0，回填的 hours[d]=0 会被前端 placeholder='0' 显示为空——视觉上看不出'已填了 0'，但 RDM 也不存 0 小时记录，所以问题不大。仅作为已知行为说明。
