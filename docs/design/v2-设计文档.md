## 0. 文档定位

本期交付的是**仿真原型**（HTML + 纯静态 JS + mock JSON），目的是让用户在写后端前先看到、点到交互细节。
本文档解释：

1. 原型由哪些页面构成、每个页面解决什么需求
2. 关键交互的设计取舍及理由
3. 与 v2.2 需求条款的逐条对应表
4. 给研发的实现注意点
5. 设计师拿不定的几处取舍，留给用户裁决

**不在范围**：后端实现、真实 RDM 对接、Python 代码、CSS 框架替换。

## 3. 页面级设计

### 3.1 登录页（对应 A1–A6）

布局沿用 v1 `templates/login.html`：紫色渐变全屏背景 + 居中白色卡片，450px 宽，圆角 20px。

#### 3.1.1 增加的元素（A6 记住账号）

在原"用户名 / 密码 / 登录"三件套**之下、登录按钮之上**，新增一组：

```
☑  记住我（用户名 + 密码）
       └ ⓘ 密码以明文形式保存在本浏览器，仅本人电脑可用

                                       [清除已保存账号]
```

- 复选框默认**勾选**；若浏览器 localStorage 已有保存账号，进入页面自动填入两个输入框（**不自动提交**）
- 提示文案紧贴复选框右下方，灰色小字 12px + 信息图标，让用户一眼看到"明文"二字
- "清除已保存账号"链接按钮放右侧，文字 link-style（无边框，紫色），点击后清 localStorage 并清空两个输入框，弹个 toast "已清除"

#### 3.1.2 错误提示（A3）

错误条用 v1 已有的 `#alertBox`，改为顶部红色横条，**不弹 alert**。文案示例：
- 用户名或密码错误
- 网络异常，请稍后重试
- 账号已被锁定，请联系管理员（占位文案，等抓包确认）

### 3.2 主页 / 工时填报表（对应 B / C / D / E / F）

#### 3.2.1 顶部周信息条（对应 D1 / D4 / C3）

```
┌────────────────────────────────────────────────────────────────────────┐
│  📅 2026 第 24 周（6/8 周一 ~ 6/14 周日）       [日期选择器▾]  ☐启用周末  │
│  ↻ 当前显示的是 RDM 中已填工时（已回填 3 项）                              │
└────────────────────────────────────────────────────────────────────────┘
```

- 主标题大字 24px："2026 第 24 周（6/8 周一 ~ 6/14 周日）"，本周以浅色 badge "本周" 标注
- **日期选择器**：用浏览器原生 `<input type="date">`（兼容好、用户熟悉），右上角。选定任意一天 → 前端用 ISO 周（周一起）算出该周，整表刷新
- **启用周末** toggle：右侧 Bootstrap switch；提示文案"启用后周六周日可编辑，切周后恢复禁用"鼠标悬停显示
- **回填提示**：若 D3 接口返回该周有已填记录，下方多一行 info 横条 "当前显示的是 RDM 中已填工时（已回填 N 项）"，背景浅蓝；若该周无记录则不显示该行

#### 3.2.2 工时表格主体（对应 B / C1 / C2 / C4 / C5）

**核心难点：每格容纳"小时数 + 完成率"两个输入，又要在 1366×768 不横向滚动。** 详见 §4。

表格列结构（共 10 列）：

| 列 | 宽度 | 内容 |
| --- | --- | --- |
| 任务 / 项目 | 220px | 任务名（粗体）+ 项目名（灰小字）+ source 标签 |
| 周一 ~ 周日 | 各 ≈ 124px | 每格 2 个输入：上半 小时数 / 下半 完成率% |
| 合计 | 80px | 该任务一周小时合计（大字深紫） |

行高 64px（足够装下两行输入），单元格 padding 6px。

##### 任务行细节

```
┌─ 任务名 ─────────────────────┬──── 周一 6/8 ────┬─ ... ─┬─ 合计 ─┐
│ 工时系统重构 (Prj-Alpha) [负责]│  [ 8 ]h            │       │  40h   │
│                              │  [80] %            │       │        │
└──────────────────────────────┴────────────────────┴───────┴────────┘
```

- 任务名 600 粗体 14px，项目名 12px `#999`
- **source 标签**用 badge，紧跟项目名右侧：
  - `responsible` → 紫底白字 "负责"
  - `participate` → 浅蓝底深蓝字 "参与"
  - `both` → 渐变底 "负责·参与"
- 小时输入：`<input type="number" step="0.5" min="0" max="24">`，宽度 64px，居中
- 完成率输入：`<input type="number" step="1" min="0" max="100">`，宽度 56px + "%" 后缀，居中
- 两个输入纵向堆叠，间距 4px，标签使用图标节省空间："🕐" 小时 / "🎯" 完成率（仅装饰）

##### 周末列样式（对应 C3）

- 默认 `disabled` + 整列背景 `#f3f4f6` + 输入框透明度 0.45
- 周末列头用 `#fff3cd` 浅黄底，写"周六 6/13"，禁用时左上角加小锁图标 🔒
- 顶部开关"启用周末"打开后：去掉锁、底色变正常、输入可点击

##### 表尾合计行（对应 C4）

```
│ 当日合计 │ 24h │ 24h │ 16h │ 8h │ 8h │ 0h │ 0h │ 本周 80h │
```

- 整行浅紫底 `#eef0fb`、字号 14px 粗体
- 每日合计 = 该列所有任务小时之和；某列总和 > 24 时数字变红 + 旁边小图标 ⚠ 警告

##### 空态

若 `/api/tasks` 返回 0 条任务（mock 也演示一种）：

```
        📭
   未找到可填写任务
状态为「未启动」或「进行中」的任务才会出现在这里
            [刷新]
```

宽 320px，紫色渐变按钮放在最上：

#### 3.2.4 提交进度面板（对应 E1 / E4）

点"提交本周工时"后，弹出一个**模态层**（不可关闭，必须等结束）。设计成"任务清单 + 进度条"形式：

```
┌─ 正在提交本周工时 ──────────────────────────────────┐
│  进度：3 / 8                                       │
│  ███████████░░░░░░░░░░░░░░░░░░  37%               │
│                                                    │
│  ✓ 工时系统重构        已提交 (40h)                │
│  ✓ 接口联调            已提交 (8h)                 │
│  ✓ 周例会              已提交 (2h)                 │
│  ↻ 文档编写            正在提交 …                  │
│  ⏳ 代码评审            排队中                      │
│  ⏳ 故障排查            排队中                      │
│                                                    │
│  请勿关闭窗口，提交过程中可能持续几十秒              │
└────────────────────────────────────────────────────┘
```

- 单任务正在提交时显示 spinner；若进入重试，文案改为"正在重试 (2/3)…"，状态色变橙
- 完成后自动关闭并弹"全部提交成功"toast；中途失败转 §3.2.5 失败弹窗

#### 3.2.5 失败弹窗（对应 E3）

任何任务 3 次重试都失败时，进度面板**变色**（顶部红色横条）并弹出失败弹窗：

```
┌─ ✕ 提交失败 ─────────────────────────────────────────┐
│                                                       │
│  任务【文档编写】（task_id: T-2026-0612）提交失败       │
│                                                       │
│  失败原因：网络异常，已重试 3 次                       │
│                                                       │
│  ⚠ 后续 2 个任务未提交，请手动到 RDM 处理：              │
│     · 代码评审                                         │
│     · 故障排查                                         │
│                                                       │
│  已成功提交的 3 个任务保持已提交状态。                   │
│                                                       │
│              [打开 RDM 网页]   [我知道了]              │
└───────────────────────────────────────────────────────┘
```

- 红边框 + 红色 ✕ 图标
- "打开 RDM 网页" 按钮直接 `window.open('http://10.111.36.3:2029', '_blank')`
- "我知道了" 关闭弹窗，把表格恢复可编辑，**保留失败任务和未提交任务的数据**，让用户能再试一次

#### 3.2.6 字段越界即时拦截（对应 F4）

不等提交，输入时实时校验：

| 触发 | 表现 |
| --- | --- |
| 小时数 > 24 | 输入框红边 + 下方红字 "单日不能超过 24 小时"，自动 clamp 到 24 |
| 小时数 < 0 | 自动 clamp 到 0 |
| 完成率 > 100 | 红边 + clamp 到 100 |
| 完成率非整数 | 自动 `Math.round()` |
| 某列合计 > 24 | 列尾合计变红 + ⚠，但不阻断提交（提示给用户自己判断） |

## 4. 关键设计决策与取舍

### 4.1 决策 D-01：每格"小时 + 完成率"的呈现方式（C2 落地）

v2.2 C2 明确"每格两个输入"，但视觉上 7 列 × 双输入很容易挤。考察了 3 方案：

| 方案 | 视觉 | 输入便利 | 屏幕占用 | 评价 |
| --- | --- | --- | --- | --- |
| A：纵向堆叠 (本设计) | 小时在上，完成率在下，行高 64px | 不需点击额外控件，一目了然 | 单元格 124×60 | 默认采用 |
| B：仅显示小时，悬停展开完成率 | 平时只 1 行，hover/focus 才出完成率 | 触摸屏不友好；漏填风险 | 单元格 124×30 | 否决（漏填风险） |
| C：完成率独立成 7 行表头下方一行 | 表格分两段：上半小时、下半完成率 | 视线跨度大，难对齐 | 占两段 | 否决 |

**结论**：方案 A。1366×768 实测每列 124px、表格总宽 1100px 左右，留出导航和操作面板后**正好不出现横向滚动**（详见 §6 实测）。

### 4.2 决策 D-02：日期选择器形态（D1 落地）

v2.2 D1 把控件形态留给设计。选择 **HTML5 原生 `<input type="date">`**：

- 优点：零依赖、无样式冲突、所有浏览器支持、可键盘输入
- 缺点：样式跨浏览器有差异（Chrome / Edge 一致，Firefox 略不同）
- 备选方案：Bootstrap 5 没有自带 date picker，引入 flatpickr 等会增加体积；不引入第三方

辅助交互：选完日期 → 前端用 ISO 周算法（`getISOWeek` 工具函数）算出周一周日 → 整表刷新 + 触发 D3 回填请求 mock

### 4.3 决策 D-03："启用周末" 开关位置（C3 落地）

v2.2 C3 把交互方式留给设计。三个候选：

| 候选 | 位置 | 选用 |
| --- | --- | --- |
| 顶部 toggle | 周信息条右上 | ✓ 采用 |
| 周末列头复选框 | 周六/周日列头各一个 | 备选 |
| 双击列头解锁 | 列头双击 | 否决（隐藏交互） |

采用顶部 toggle，理由：①统一控制周六周日两列、②比列头复选框醒目、③切周后开关自动复位
（与 v2.2"切下一周后默认状态恢复禁用"一致）

### 4.4 决策 D-04：source 标签的展示

v2.2 §3.1 A2 说 `source` 字段"前端可选择展示"。设计选择**始终展示**，作为任务行的小 badge：

- 让用户一眼区分"这是我负责的还是仅参与的"，有助于优先填写"负责"的任务
- 不做切换 / 过滤（v2.2 A6 明确"不暴露 scope 切换器"）

### 4.5 决策 D-05：提交进度面板不可关闭

提交是串行的、失败后用户要看清失败在哪、未提交的是哪些。如果允许中途关闭面板，用户可能错过失败弹窗。
所以面板做成 modal 且**没有 close 按钮**，必须等过程结束（成功 / 失败弹窗）才能关闭。

### 4.6 决策 D-06：mock 数据要演示的所有形态

mock JSON 必须覆盖以下分支，让用户在原型里能完整体验：

| 形态 | mock 怎么造 |
| --- | --- |
| 多种 source | 准备 3 个 responsible、2 个 participate、1 个 both |
| B4 过滤 | mock 中故意混入 2 个 closed / 1 个 done，证明被过滤不进表 |
| D3 回填 | 切到"上一周"返回 3 项已填记录，切到"本周"返回 0 项 |
| E3 失败 | 任务 ID 含 `fail` 的串行提交时模拟 3 次失败 |
| 空态 | 切到"未来某一周"返回空任务列表 |
