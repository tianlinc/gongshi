## 1. 需求目标

把 `app.py` 中 `RDMClient` 的两个 stub 方法替换为对 RDM (`http://10.111.36.3:2029`)
的真实接口对接，让"周工时填报"全流程在生产可用：

- `get_my_tasks()` —— 真实拉取当前用户的任务列表（替换 3 条硬编码示例）
- `submit_work_log()` —— 真实把一周工时写入 RDM（替换只 `print()` 的占位）

## 2. 范围

### 做（in scope）

- `get_my_tasks()` 的真实实现，**只覆盖"我负责的任务"这一种 scope**（其他 3 种延后）
- `submit_work_log()` 的真实实现，能把整周 7 天 × N 任务的工时写到 RDM
- 前端 `templates/dashboard.html` 改造：字段统一 snake_case、提交 payload 必带
  `task_id`、未登录响应自动跳 `/`
- 失败路径（登录过期、网络错误、字段校验失败）返回明确的中文 message

### 不做（out of scope）

- **session 持久化重构** —— Flask 重启后用户重登的现状保留（用户决定）
- RDM base URL 配置化 —— 仍硬编码
- 工时编辑 / 删除 / 历史查询 —— 只做"看任务 + 提交本周"
- 引入测试框架、CI、lint —— 项目无此约定
- 改动 `archive/` 老版代码 —— 只参考，不动
- 移动端适配、UI 改版

## 3. 子需求与验收标准

### 3.1 子需求 A：拉取"我的任务"列表

替换 `RDMClient.get_my_tasks()`。**本期只做"我负责的任务"这一种 scope**——
前端不加 scope 切换器，其他 3 种 scope（我参与 / 我分配 / 待我审核）延后处理。

**验收标准**：

- A1：登录后调 `GET /api/tasks`，返回 `{success: true, tasks: [...]}`，数量与该
  用户在 RDM 网页"我负责的任务"radio 下看到的任务数一致。
- A2：每个 task 至少含 `id`（RDM 内部真实标识）、`name`、`project` 三个字段。
- A3：登录态失效时，后端返回 `{success: false, message: "未登录"}`（或可识别等价
  消息）；**前端 dashboard.html 检测到此响应必须自动跳 `/`，不弹框、不悄悄失败**
  （见 A6）。
- A4：代码里不再出现 `TASK001 / TASK002 / TASK003` 等硬编码示例。
- A5：本期 `get_my_tasks()` 的实现可以不接受 scope 入参（或固定走"我负责"），
  无需暴露 scope 切换。其他 3 种 scope 不在本期 AC 范围内，不验。
- A6（新增，跨 A/B 适用）：**所有 `/api/*` 路由返回未登录响应时，前端必须自动
  跳 `/`**——dashboard 上不弹任何对话框、不留在原页面带空数据。该跳转一次性
  覆盖 `/api/tasks` 和 `/api/submit-timesheet` 两条路径。

### 3.2 子需求 B：提交本周工时

替换 `RDMClient.submit_work_log()`。

**验收标准**：

- B1：前端填好工时点提交后，到 RDM 工作日志页能看到本周对应任务的新记录，每天
  小时数与前端一致。
- B2：completion_rate 能真实写入 RDM，事后在 RDM 上可见。
- B3：提交失败时返回 `{success: false, message: "..."}`，message 为人类可读中文；
  **未登录场景同 A3/A6：前端必须自动跳 `/`，不弹框**。
- B4：同一周二次提交的行为（覆盖 / 追加 / 报错）必须**明确且可观测**——不允许
  "默默成功但什么都没写"。具体策略见 §4 Q2 确认后补充。
- B5：保留提交内容的 stdout 日志（或换成 logging），便于线上排查。

### 3.3 子需求 C：前后端契约对齐

**决定（用户拍板）**：

- 统一使用 **snake_case** 命名
- payload **必须包含 `task_id`**（RDM 任务真实标识）

**验收标准**：

- C1：前端 `templates/dashboard.html` 需修改：
  - `weekStart` → `week_start`
  - `completionRate` → `completion_rate`
  - 提交 payload 中每个 task 项必须带 `task_id`（取自 `/api/tasks` 返回的 `id`），
    不能只发 `name`
- C2：后端 `app.py` 中所有解析处使用 snake_case 字段，与前端一致。
- C3：契约对齐后，前端发送的字段集合与后端期望字段集合一致——在 review 阶段
  对照确认，禁止"前端发了后端没读"或"后端要的字段前端没发"。

## 5. 验收方式

研发工程师交付后，由 requirements-analyst 按 §3 中所有 A* / B* / C* 验收项逐条核对：

1. 本地 `python app.py` 跑起来，浏览器走完"登录 → 看任务列表 → 填工时 → 提交"全流程
2. 在 RDM 网页上确认提交结果真实落库
3. 制造失败路径（断网、改 cookie、空提交）验证错误 message 符合 B3 / A3
4. 通过后在本文件追加 `## 6. 验收记录` 段，写日期、通过项、遗留问题
5. 任意一项 AC 不通过则退回研发，**不可绕过**
