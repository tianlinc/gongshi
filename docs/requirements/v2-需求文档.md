## 0. 用户原话

> 基于用户提供的账号、密码，自动登录工时系统，然后从【我的任务】菜单下的【我负责的任务】、
> 【我参与的任务】里动态找到可以填写的任务。用户可以批量填写每个任务每一天的投入工作量
> （小时）和完成率（百分比），然后一起提交。用户一般 1～2 周统一填一次，通常只填工作日。

本文档把这段话拆成可验收的子需求；用户对 7 个原始模糊点的拍板见 §7。

## 1. 范围

### 做（in scope）

- 登录鉴权：用户填 RDM 账号密码，本系统代为登入 RDM；**记住上次的用户名 + 密码**（明文 localStorage，可随时修改/清除）
- 任务拉取：动态拉【我负责的任务】+【我参与的任务】
- 任务过滤：**只保留状态 ∈ {未启动, 进行中} 的任务**
- 工时输入：任务 × 7 天的批量填写界面，单元格输入小时数
- 完成率输入：**每个任务的每一天都有独立完成率**（不是任务级总值）
- 工作日规则：默认仅工作日可填，**周末可手动启用**
- 跨周/历史周：**日期选择器自由选**任意日期，自动定位该周；**回填该周 RDM 已存在的小时数和完成率**
- 提交：**逐任务串行提交**，失败自动重试 3 次后报错；失败后停止剩余任务、弹窗提示用户手动处理
- 错误反馈：登录失败、未登录、网络错误、字段非法 → 中文消息
- 二次提交：**覆盖** RDM 中已有记录

### 不做（out of scope）

- "我分配的任务"、"待我审核的任务" 两种 scope
- 工时编辑/删除/历史查询（提交后改回 RDM 网页改）
- 一次性事务回滚（用户已拍板"失败处停止 + 用户手动"，不做自动回滚）
- 移动端 / 主题 / 多人协作
- 测试框架 / CI / lint
- 改动 `archive/` 老版代码
- session 持久化（Flask 重启需重登；与 v1 一致）
- RDM base URL 配置化（仍硬编码 `http://10.111.36.3:2029`）

## 2. 子需求与验收标准

### 2.1 子需求 A：登录鉴权 + 记住账号

用户在本系统首页填 RDM 账号密码，本系统调用 RDM 登录接口完成登录。CLAUDE.md 中
记录的"双重编码 + AES-ECB（密钥 `abcdefgabcdefg12`）"加密流程仍然适用。

**验收标准**：

- A1：未登录访问 `/` 显示登录页，登录页有用户名、密码、登录按钮三件套。
- A2：填正确账号密码（例如 §4 测试账号）点登录 → 后端调 RDM 成功 → 浏览器进入填报主页。
- A3：填错账号或密码 → 留在登录页 + 中文错误提示（如"用户名或密码错误"），**不弹原生 alert**。
- A4：登录态失效（cookie 被清、Flask 重启 client dict 丢失等）时，任何 `/api/*` 调用
  必须返回 `{success: false, message: "未登录"}`，前端识别后**自动跳 `/`，不弹框**。
- A5：登出按钮清 session、回到登录页（**但 localStorage 中记住的账号密码不被清除**——见 A6）。
- **A6（已拍板，Q6）**：登录页**记住上次的用户名 + 密码**——
  - 实现方式：浏览器 `localStorage`（前端本地存储，不上服务器以外的地方）
  - 默认勾选"记住我"复选框；用户可取消勾选 → 下次不存
  - 登录页提供"**清除已保存账号**"按钮，一键清掉 localStorage 里的用户名密码
  - **必须在登录页明确文案告知用户："密码以明文形式保存在本浏览器，仅本人电脑可用"**，让用户自己决定要不要勾
  - 进入登录页时，如有保存的账号密码，**自动填入两个输入框**（不自动提交，留给用户点登录）

### 2.2 子需求 B：拉取"我负责"+"我参与"任务（按状态过滤）

替换原 `RDMClient.get_my_tasks()`。同时拉两类、合并去重、**按状态过滤**。

**验收标准**：

- B1：登录后调 `GET /api/tasks`，返回 `{success: true, tasks: [...]}`。
- B2：tasks 数量等于该用户在 RDM 网页"我负责的任务" + "我参与的任务" 两个 radio
  下任务数 - 两边重复数 - 状态不在 {未启动, 进行中} 中被过滤掉的数（见 B4）。
- B3：每个 task 至少含：
  - `task_id`（RDM 内部真实标识，提交时必带）
  - `name`（任务名）
  - `project`（所属项目）
  - `status`（任务状态，至少能映射出"未启动" / "进行中" / 其他）
  - `source` ∈ {`"responsible"`, `"participate"`, `"both"`}
  - 其他 RDM 字段（如计划起止时间、当前完成率等）按 R1 抓包补齐
- **B4（已拍板，Q1）**：**只保留 `status` ∈ {"未启动", "进行中"} 的任务**。其他状态
  （已完成 / 已关闭 / 审核中 / 已取消 等）**全部过滤掉**，不进 `/api/tasks` 返回。
  - RDM 状态字段的真实英文/中文取值由研发抓包确认后写入 R5
  - 验收时人工对照：RDM 网页两个 tab 下手数状态为"未启动"或"进行中"的任务数，应等于
    `/api/tasks` 返回去重后的条数
- B5：去重——同一 `task_id` 在两边都出现，合并成一条，`source` 标 `"both"`。
- B6：代码里不能再有 `TASK001/002/003` 之类硬编码示例。

### 2.3 子需求 C：每日工时 + 每日完成率批量输入

前端表格：行是任务、列是 7 天，**每一天的格子里同时含小时数 + 完成率两个输入**。

**验收标准**：

- C1：每个工时单元格可输入数字，**步进 0.5 小时**，范围 0～24（依赖 R3 实测）。
- **C2（已拍板，Q2）**：完成率粒度 = **每任务每天各一个**。
  - 表格列结构改为：「任务名 / 项目 | 周一(h, %) | 周二(h, %) | ... | 周日(h, %) | 合计」
  - 每个单元格内有**两个子输入**：小时数（步进 0.5，0～24）+ 完成率（0～100 整数）
  - 完成率每天独立：用户可以填周一 80%、周二 85%、周三 90%……
  - 当天小时数为 0 时，该天的完成率可以为空（不参与提交）
  - 具体视觉/交互细节交 product-designer 设计
- **C3（已拍板，Q4）**：周末两列**默认置灰禁用**；表格顶部或控件区**提供一个"启用周末"
  开关**——用户勾选后，周六周日列变为可输入。
  - 启用周末仅影响当前选中的周；切换到下一周后默认状态恢复为禁用（不持久化用户偏好）
  - 具体开关位置、文案、交互交 product-designer
- C4：表格每天列底部显示当日合计小时数、每行尾显示该任务一周合计小时数。
- C5：日期显示要带年月日（"6/8 周一"），避免跨周误填。

### 2.4 子需求 D：日期选择器 + 跨周回填

用户用日期选择器选任意一天，系统自动定位到那一周；切到旧周时**回填该周 RDM 中已存在
的数据**。

**验收标准**：

- **D1（已拍板，Q5）**：周切换控件 = **日期选择器**（任意日期，无范围限制）。
  - 用户选某一天 → 系统自动算出该日期所在的 ISO 周（周一为起始）→ 展示整周
  - 默认值 = 今天（即默认进入本周）
  - 控件具体形态（原生 date picker / Bootstrap date picker / 其他）交 product-designer
- D2：默认显示"本周"（含今天的那个 ISO 周）。
- **D3（已拍板，Q5b）**：切到任意一周时，**调用 RDM 工时查询接口回填**该周已存在的
  小时数和每日完成率。
  - 调用接口为新增的 `GET /api/timesheet?week_start=YYYY-MM-DD`
  - 后端走 RDM 查询接口（具体端点由 tech-architect 调研，纳入 R6）
  - 返回结构与提交 payload 对称：`{success: true, tasks: [{task_id, hours[7], completion_rates[7]}]}`
  - 前端拿到回填数据，按 `task_id` 匹配当前任务列表，把对应行的小时数和完成率填进去
  - 找不到匹配 task_id 的回填记录（如 RDM 里有但 B4 过滤掉了）→ 忽略，不报错
- D4：界面上要清楚显示当前操作的是哪一周（如"2026 第 24 周（6/8 ～ 6/14）"）。

### 2.5 子需求 E：串行提交 + 失败重试 + 弹窗提示

**一次提交 ≠ 一次性 POST**。用户拍板"一个一个提"，失败处停止剩余任务、弹窗提示。

**验收标准**：

- **E1（已拍板，Q3/Q7）**：提交流程改为**逐任务串行 POST**——
  - 前端把当前周"有数据的任务"（至少一天小时数 > 0）按列表顺序排队
  - 每次只 POST 一个任务到后端 `/api/submit-timesheet`（一次一个 task 的 payload）
  - 后端按收到的 task 写入 RDM，**返回 RDM 真实写入结果**（不假装成功）
  - 前端拿到该任务 success 后，继续下一个；拿到 fail 后进入 E3 重试逻辑
  - **二次提交策略 = 覆盖**（已拍板 Q7）——后端调 RDM 写入接口时，若该任务该周已存在
    记录，要执行覆盖。具体覆盖策略（先删后插 / update / RDM 提交接口本身支持）由
    tech-architect 调研，纳入 R7
- E2：单任务 payload 字段命名 snake_case，必含：
  ```json
  {
    "week_start": "2026-06-08",
    "task": {
      "task_id": "...",
      "name": "...",
      "hours": [8, 8, 8, 8, 8, 0, 0],
      "completion_rates": [80, 85, 90, 95, 100, null, null]
    }
  }
  ```
  - `hours[]` 长度固定 7（周一→周日），未填位置 = 0
  - `completion_rates[]` 长度固定 7，对应位置当日小时数为 0 时该位置 = `null`（不写入 RDM）
- **E3（已拍板，Q3）**：单任务提交失败时**自动重试 3 次**——
  - 重试间隔建议 1s / 2s / 4s 指数退避（具体由 tech-architect 决定）
  - 3 次都失败 → **停止后续任务的提交**（不继续排队任务）
  - **前端弹窗**：内容必须包含
    1. 失败任务名 + task_id
    2. 失败原因（后端返回的 message）
    3. 一句明确提示："**任务【X】提交失败，后续 N 个任务未提交，请手动到 RDM 处理**"
    4. 列出后续未提交的任务名单
  - 弹窗确认后，**已成功提交的任务保持已提交状态**（不回滚）
- E4：提交过程中前端必须显示**实时进度**——"正在提交 3/8：任务 X"，让用户看到串行进度
  ，避免无反馈。
- E5：未登录场景按 A4 跳登录页（不弹失败弹窗，直接跳）。
- E6：保留每次 RDM 写入的 logging 日志（不仅 print），含 task_id / 重试次数 / 最终结果，
  方便事后排查。

### 2.6 子需求 F：错误反馈

横切所有接口的错误处理标准。

**验收标准**：

- F1：所有 `/api/*` 接口失败返回 `{success: false, message: "中文人类可读消息"}`。
- F2：登录态失效统一用 `"未登录"` 文案（或可识别等价消息），前端识别后跳 `/`。
- F3：网络/超时类错误前端展示"网络异常，请稍后重试"，不要把 Python Traceback 抛到 UI。
- F4：字段校验类错误（小时数超 24、完成率非整数等）应在前端拦截，不发请求。

## 3. 验收方式

研发交付后，由 requirements-analyst 按 §2 中所有 A* / B* / C* / D* / E* / F* 逐条核对：

1. 本地 `python app.py` 跑起来
2. 用 §4 测试账号登录，走完"登录 → 看任务表 → 用日期选择器切到上一周 → 看回填数据
   → 在本周改填 → 提交"全流程
3. 在 RDM 网页核对：
   - "我负责 + 我参与" 中状态为 {未启动, 进行中} 的任务条数与 `/api/tasks` 吻合（B2、B4）
   - 提交后工作日志页能看到对应记录，每天小时数 + 每天完成率与前端一致（E1）
   - 二次提交同一周，看 RDM 是否真覆盖（E1 覆盖）
4. 制造失败路径：
   - 断网或 mock 后端报错 → 验证重试 3 次 + 弹窗内容（E3）
   - 改 cookie → 验证未登录跳转（A4、F2）
   - 字段越界 → 前端拦截（F4）
   - 关掉浏览器后重开 → 验证 localStorage 自动填账号密码（A6）
5. 通过后追加 §8 验收记录
6. 任一项不过则退回研发，不绕过
