## 0. TL;DR

- **不沿用 `app.py` 现状**，但保留 `RDMClient` 类名 + Flask + 单进程 + 双模板 的整体形态。按 ADR-0001 决策 B 的 5 模块布局做一次实质拆分（决策 A/C 同期落地），原因：本期要接的真实接口比 stub 复杂一个数量级，再叠在 285 行单文件里写不下。
- **RDM 工时填报的真实路径是 `pages/myspace/log/workLogView.jsf`**（不是 `workLogList.jsf`，那是别人的日志列表）。需要先以"周起始日"为 key GET 一次取 JSF ViewState + 已存在的 `reportId`，再 A4J POST `operate:saveLog` 写入。
- **任务列表来自 `myTask.jsf` 的 A4J 局部刷新**，响应是 XML 包裹的 HTML 表格，需要解析 `<tr id=...>` 取 task_id；同时也要 GET workLogView 拿"本周可填的任务"（包含计划内 + 已建过的计划外）。
- **工时数据用"计划外任务"(`_unplannedInfo`) 路径**写入——这是 RDM 里"用户主动声明今天在某任务上花了 X 小时"的渠道；序列化格式已逆出，见 §4.3。
- **session 决策**：用 ADR-0001 决策 A 的"凭据回放"——浏览器 cookie 存加密密码，进程重启自动重登；RDM session 本身因为短寿且不可控，不持久化。

## 1. 当前状况 vs 需要支撑的目标

| 维度 | v1 现状 | v2 目标 |
| --- | --- | --- |
| 文件结构 | 单文件 `app.py` 285 行，全部业务 + 路由 + 全局 dict 混在一起 | 拆 5 模块（client / parsers / routes / session_store / app 入口） |
| 任务接口 | stub 写死 3 条数据 | 真实拉两个 scope（`myRecive` + `myReciveActor`），去重合并，按 Q1 规则过滤 |
| 工时接口 | stub `print()` 一下返回成功 | 真实写入 RDM 工作日志（走 `workLogView.jsf` 的 `unplannedInfo` 路径） |
| 周切换 | 没有，前端固定本周 | 按 Q5 支持上 N 周；切换时后端要重新 GET workLogView 取该周的 ViewState/reportId |
| 完成率 | 前端有字段，后端忽略 | Q2 = 单任务总体 → 拼到 `_rate{day}`；Q2 = 每天独立 → 仍然是 `_rate{day}`，前端拆字段 |
| Session | 进程内 dict，热重载即掉 | 浏览器 cookie 存加密密码，自动重登；`/api/*` 401 → 前端跳登录 |
| 错误反馈 | 200 + `{success:false}` | 业务错误维持 200；登录态失效改 401（满足需求文档 F2 / A4） |

## 2. 调研产出：RDM 真实接口实证

完整证据见 `tools/captured/`。下列结论都是**实跑 + 静态分析双确认**得到的，不是从 `tools/myTask.html` 文档级猜测。

### 2.1 关键端点清单

| 用途 | URL | 方法 | 关键参数 |
| --- | --- | --- | --- |
| 登录 | `/j_security_check` | POST form | `j_username`/`j_password`（base64 → AES-ECB → base64） |
| 主页（拿菜单/导航） | `/main.do` | GET | 登录后重定向到此 |
| 任务列表（页面壳） | `/pages/task/list/myTask.jsf` | GET | 返回含 `javax.faces.ViewState` 的 HTML，表格体为空，需 A4J 二次请求 |
| 任务列表（数据） | `/pages/task/list/myTask.jsf` | POST | A4J 局部刷新，`operate:refreshBody=operate:refreshBody` + `cate=0/22/1/2` 切 scope（0=我负责、22=我参与、1=我分配、2=待我审核） |
| 工时填报页（数据） | `/pages/myspace/log/workLogView.jsf` | GET | `?isView=N&userId=&reportId=&startDate=YYYY-MM-DD` 返回服务端预渲染的整周 JSON（藏在 `<input id="data">` 里）+ ViewState |
| 工时提交 | `/pages/myspace/log/workLogView.jsf` | POST | A4J，`operate:saveLog=operate:saveLog` + 完整表单字段（见 §4.4） |
| 服务器今天日期 | DWR `jtaskEntityBean.dynGetServerToday` | POST | 切周时用来判断"本周" |

**菜单"工作日志"对应的 `workLogList.jsf` 不是填工时页，是别人的日志列表（关注 / 下属 / 全部）**。这是一个容易踩错的点——v1 的 stub 注释里就误判了。真正的入口：从 myTask 双击任务行或 workLogList 双击某条记录 → 打开 `workLogTab.jsf?reportId=...&userId=...` → 这个 tab 容器里 iframe 嵌入 `workLogView.jsf`。本期我们不走"双击行"，直接构造 workLogView 的 URL 即可。

### 2.2 登录复用（不变）

CLAUDE.md 里的双层编码 + AES-ECB 流程**实测有效**，但需要补一个新发现：**`requests.Session` 必须带浏览器风格的 `User-Agent`**，否则 RDM 服务器随机断开（实测 `ConnectionResetError`），v1 没有遇到是因为 v1 没真正调用 `/api/*` 之外的页面。

```python
session.headers.update({
    'User-Agent': 'Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120 Safari/537.36',
    'Accept-Language': 'zh-CN,zh;q=0.9',
})
```

### 2.3 任务列表 A4J 响应解构（R1 实证答案）

请求示例（POST `/pages/task/list/myTask.jsf`，content-type `application/x-www-form-urlencoded`，需要带 `Faces-Request: partial/ajax` + `X-Requested-With: XMLHttpRequest`）：

| 字段 | 含义 | 备注 |
| --- | --- | --- |
| `operate` | 表单 marker | 固定 `operate` |
| `operate:refreshBody` | 触发 action | 值与 key 同名（A4J 约定） |
| `javax.faces.ViewState` | JSF 状态 | 从前一次 GET 取，**不能缓存**（每次刷新都会变） |
| `cate` | scope | `0`=我负责，`22`=我参与，`1`=我分配，`2`=待我审核 |
| `type` / `module` | 固定 | `TSK` |
| `refreshType` | 0=分页刷新，1=全量 | 第一次拿数据用 0 |
| `isInit` | `Y` | 首次进入页面置 Y，否则会带上次的过滤器 |
| `page` | 页号 | `1` 起 |
| 其余 hidden | 见 `tools/myTask.html` 第 300~340 行 | 不传可能 NPE，传空串即可 |

响应是 `<?xml version="1.0"?>` 包 HTML 片段，每条任务一行：

```html
<tr class="body-row" id="{taskId-uuid}" status="RN" ...>
  <td><div class='sn-div' item='1'>1</div></td>          <!-- 序号 -->
  <td><img src="/images/icon072.gif" /></td>             <!-- 状态指示灯 -->
  <td></td>                                              <!-- 关注 -->
  <td><span>18</span></td>                               <!-- 优先级 -->
  <td></td>                                              <!-- 标签 -->
  <td>...<span class='item-span auto-link'><a>{任务名}</a></span>...</td>
  <td>{完整路径}</td>                                    <!-- 全路径 -->
  <td><a onclick="openEntity('{projectId}', 'PJT')">{项目名}</a></td>
  <td>{所属计划}</td>
  <td>2026-06-01</td>                                    <!-- 计划开始 -->
  <td>2026-06-30</td>                                    <!-- 计划结束 -->
  <td></td>                                              <!-- 实际开始 -->
  <td></td>                                              <!-- 实际结束 -->
  <td>{责任人}</td>
  ...
</tr>
```

解析要点：

- `tr.id` 就是 `task_id`（uuid），提交时必带——对应需求 B3
- `tr[status]` ∈ `{RN, NS, FN, ...}`：`RN`=进行中、`NS`=未开始、`FN`=完成——**Q1 的过滤规则建议**：只保留 `RN` + `NS`，排除 `FN`/`SP`(暂停)/`FS`(强制终止)；最终以用户回答 Q1 为准
- 列顺序固定但**不一定每个 td 有内容**，必须按位置取，不要按 class 取
- 字符编码是 GBK（v1 已知陷阱），不指定 `r.encoding = 'utf-8'` 中文乱码

### 2.4 工时填报数据模型（R2 实证答案）

GET `/pages/myspace/log/workLogView.jsf?isView=N&userId=&reportId=&startDate=2026-06-08`（startDate 必须是该周周一）。响应有两块关键内容：

1. **`<input id="data" value="...">`** —— 服务端预渲染的整周数据 JSON（HTML 实体编码），结构：

```json
{
  "logHead": [...列定义],
  "logBody": [],                  // 已填工时的任务记录（编辑现有日志时非空）
  "workingDay": [1,1,1,1,2,0,0],  // 周一~周日工作日标记：1=工作日,2=今天,0=周末
  "unHead": [...计划外任务列定义],
  "unBody": [],                   // 已添加的计划外任务
  "weekHead": [...下双周计划列],
  "weekBody": [],
  "finishHead": [...本周已完成列],
  "finishBody": []
}
```

2. **隐藏字段一批**（form `#operate` 内）：

```
_startDate (=startDate)        _objectId (=reportId, 新建时为空)
_summary                       _workRemark / _issueRemark
_rightUserId                   _unplannedInfo  <-- 本期核心
loginUser (uuid)               dptId / dptName
javax.faces.ViewState          isThisWeek (Y/N)
```

3. **触发器**（A4J 链接）：

```
operate:saveLog          ←本期主要用这个，提交工时
operate:refreshLog       ←切周后用 GET + refreshLog 重渲染
operate:refreshData      ←仅刷新任务记录
operate:refreshDaily     ←saveLog 完成回调用，刷新日报
operate:reloadLog        ←重新载入整页
operate:refreshNote      ←刷批注
operate:refreshDataForRole ←切换"我负责/我参与"双周计划
```

### 2.5 `_unplannedInfo` 序列化格式（**本期最关键的逆向产物**）

来源：`scripts/myspace/unplannedTask.js` 的 `buildUnInfo()` + `composeUnInfo()`。

字符串由 4 个 delimiter 编织而成：

```python
S1 = "#-@%!-@#"        # 字段间 / 段间分隔符（同一行内）
S2 = "#-%@-#@"         # 字段名与值的分隔符
S3 = "#-%#!#-@-#@"     # 任务行之间的分隔符
S4 = "#-%#!A-#-@-#@"   # 字段名 与 day 值的分隔符
```

一条任务的序列化字符串（其中 `{day}` 是周几索引 0~6，多个 day 用逗号串联在行 `day` 属性里）：

```
{taskId or ' '}{S1}{day-csv-trimmed}
  [{S1}{day}{S2}_rate{S4}{rate_value}{S2}_effort{S4}{effort_value}{S2}_remark{S4}{remark}{S2}_issue{S4}{issue} ...]   # 每个 day 重复一遍
  [{S1}{fieldName}{S2}{value} ...]                                                                                     # 自定义字段（Name/ObjectID/TypeID 等）
```

多条任务用 `{S3}` 连接，最后整体塞进 `_unplannedInfo` 隐藏字段一次性 POST。

**这意味着**：本工具不需要"每个任务每天发一次请求"——一周一次性写入。这对 RDM 服务器友好，也容易实现 E1 的"批量"语义。

### 2.6 提交时的字段最小集（saveLog）

POST `/pages/myspace/log/workLogView.jsf`，参数：

```
operate=operate
operate:saveLog=operate:saveLog
javax.faces.ViewState=...
startDate=2026-06-08
isView=N
userId=
isOwner=Y
objectId=                  # 新建时空，已存在则填 reportId
summary=                   # 工作总结（HTML，可空）
workRemark=                # 7 天工作情况，用 #-%@-#@ 分隔
issueRemark=               # 7 天存在问题，同上
rightUserId=
unplannedInfo=<§2.5 拼出的整串>
loginUser=<用户 uuid>
data=<透传 GET 时拿到的 data 字段>
```

**注意 `workRemark` / `issueRemark` 的格式**（来自 workLogView.js 的 `buildDetailInfo()`）：每条 textarea 的内容用 `#-%@-#@` 分隔，空内容用单空格占位。一周 7 天 = 7 段。这两个字段是"工作情况总结"用的，本工具默认填 7 个空格段（用户在 RDM 网页二次补充），不强制要求用户填。

### 2.7 时间日期处理

- 服务器今天用 DWR `jtaskEntityBean.dynGetServerToday()` —— 但 DWR 请求构造略繁琐，本期**用客户端 `datetime.now()` 即可**，时差风险很小（同一内网）
- "周起始日"用 ISO 周一：`week_start = today - timedelta(days=today.weekday())`，v1 已有，保留
- workLogView 收到的 `startDate` 必须是周一 `YYYY-MM-DD`，否则会被服务端 normalize 到该周的周一

**`web/session_store.py`** —— ADR-0001 决策 A：

```python
clients: dict[str, RDMClient] = {}

def get_or_revive_client() -> RDMClient | None:
    u = flask.session.get('username')
    if not u: return None
    if u in clients: return clients[u]
    enc_pwd = flask.session.get('enc_pwd')
    if not enc_pwd: return None
    c = RDMClient(config.BASE_URL, config.AES_KEY)
    r = c.login_with_encrypted(u, enc_pwd)
    if not r['success']:
        flask.session.clear()
        return None
    clients[u] = c
    return c
```

### 3.4 前后端分工

- **后端做**：登录、RDM 通信、_unplannedInfo 序列化、task 过滤、错误翻译为中文
- **前端做**：周表格 UI、本地数据校验（C1/F4：小时数 0-24 步进 0.5，完成率 0-100 整数）、401 拦截跳登录、周切换控件
- **不存任何前端状态到后端**：前端是无状态的"输入面板"，每次 submit 把整周数据完整发给后端；后端不缓存周数据（缓存了反而会因 Q5b 切周变复杂）

## 4. 关键技术决策

### 4.1 不引入 DWR 客户端

虽然 RDM 大量用 DWR（`jtaskEntityBean.*` 等），但本期需要的两个功能（任务列表 + 提交工时）都能走 JSF/A4J 完成。引入 DWR 客户端要解决：协议编码、回调序列、批处理 ID 维护——成本不值得。`dynGetServerToday` 用客户端时间代替（§2.7）。

### 4.2 ViewState 不缓存

JSF 的 `javax.faces.ViewState` 每次 A4J 响应都会返回新值。我们的策略是：**每次组合操作（拉任务 / 提交工时）都先 GET 一次拿到最新 ViewState**，多 1 个 HTTP 来回换稳定性。`RDMClient` 内部不维护 ViewState 缓存。

### 4.3 选用"计划外任务"路径写工时，而不是"计划内任务"

`workLogView` 的"工作任务记录"区（`taskRecordDiv`）展示的是 RDM 项目计划里已分解到本人的任务节点——这些**用户不能在这里直接填工时**，只能看见汇总。真正可输入的是下方的"**计划外任务**"区（`unplannedTab`），它的语义是"我今天/这周在某任务上花了 X 小时"，提交后 RDM 自动汇入对应任务的工时统计。这恰好是本工具想要的语义：从 myTask 拉到的任务条目，都作为"计划外任务"补充提交。

**风险**：若用户的某任务 RDM 强制走计划内分解节点的工时报告流程（部分项目可能配置），用计划外路径写入会触发服务端校验失败。属于 Q1 未答前的灰色地带，本期假设：所有从 `myTask.jsf` 能拉到的任务都允许走计划外写入。E3 验收时若发现某任务写不进，回到本节修订。

### 4.4 GBK 编码处理

实测 RDM 大量页面在 `Content-Type` 里没声明 `charset`，requests 默认按 ISO-8859-1 解码。解决：**在 `RDMClient` 内对所有响应统一 `r.encoding = r.apparent_encoding or 'gbk'`**——比逐处指定省事。这也是 v1 没踩到但 v2 会大量踩到的坑（v1 stub 不解析 HTML）。

### 4.5 RDM 5xx / connection reset 的重试策略

实测在没设置 User-Agent 时服务器会 reset；即使设了，长会话也偶发 reset。**所有对 RDM 的 HTTP 请求加一层 retry**（最多 2 次，间隔 1s），仅对 connection-level 错误重试，HTTP 4xx/5xx 不重试。放在 `RDMClient._request()` 这个内部方法里，业务方法不感知。

### 4.6 不引入异步

`/api/submit-timesheet` 同步执行（GET workLogView → POST saveLog），最坏 2~3 秒。需求"批量"+"用户 1~2 周一次"决定了并发极低，不需要 async / 队列。

### 9.2 Q5b 修订：旧周回填已存在的工时

### 9.3 Q3 修订（**最大的架构调整**）：逐任务串行 + 失败重试 + 失败后停止

**v2.0 假设**：`submit_work_log` 一次性把整周所有任务编码到 `_unplannedInfo` 一次 POST。
**v2.1 用户决定**：逐任务串行，每个任务失败重试 N 次后仍失败则停止，已成功的保留，返回逐任务结果给前端弹窗。

**RDM 端限制确认**：`saveLog` 是整批接口，一次 POST 写整周所有计划外任务，**RDM 不返回逐任务结果**——这意味着 v2.0 的批量路径**无法满足 Q3 的"逐任务报告失败"语义**。

**架构变更**：

```python
# rdm/client.py
@dataclass
class SubmitResult:
    succeeded: list[str]                      # 已成功提交的 task_id 列表
    failed_task_id: str | None                # 卡住的那个任务（None = 全部成功）
    failed_reason: str | None                 # 中文失败原因
    remaining: list[str]                      # 因 failed 而未尝试的 task_id 列表

def submit_work_log(self, week_start: date,
                    tasks: list[TaskEntry],
                    retry_per_task: int = 2) -> SubmitResult:
    """逐任务串行 POST，失败重试 N 次仍失败则停止。
    每个任务都重新 GET workLogView 拿最新 ViewState + reportId。
    第一个任务提交后，reportId 由 RDM 创建；后续任务带这个 reportId 走"覆盖"路径。"""
    succeeded = []
    for i, entry in enumerate(tasks):
        for attempt in range(retry_per_task + 1):
            try:
                ctx = self.get_week_context(week_start)
                form = build_save_form(week_start, ctx.view_state,
                                       ctx.report_id, [entry],   # ★ 注意：单任务列表
                                       ctx.login_user_id, ctx.data_json)
                ok = self._post_save(form)
                if ok:
                    succeeded.append(entry.task_id)
                    break
            except Exception as e:
                if attempt == retry_per_task:
                    return SubmitResult(succeeded=succeeded,
                                        failed_task_id=entry.task_id,
                                        failed_reason=str(e),
                                        remaining=[t.task_id for t in tasks[i+1:]])
        else:  # for-else: 重试全部失败
            return SubmitResult(succeeded=succeeded,
                                failed_task_id=entry.task_id,
                                failed_reason="提交失败（已重试 N 次）",
                                remaining=[t.task_id for t in tasks[i+1:]])
    return SubmitResult(succeeded=succeeded, failed_task_id=None,
                        failed_reason=None, remaining=[])
```

**API 响应形状变更**（§7）：

```typescript
type SubmitResp = {
  success: boolean,                  // 全部成功 = true；否则 false
  succeeded_task_ids: string[],
  failed_task_id?: string,
  failed_reason?: string,
  remaining_task_ids: string[],
}
```

**前端含义**：
- 整体 success 时简单弹"提交成功 N 条"
- 部分失败时弹窗列出：成功了 X 条 / 失败在哪个任务（显示任务名）/ 还有 Y 条没提交；按钮提供"重试剩余"（前端只发剩余 + 失败的那些 tasks 再调一次）
- 后端**不主动重试整批**——retry 仅限单任务内 N 次

**代价说明**：N 个任务 → N 次 GET workLogView + N 次 POST saveLog。考虑 1 周一般 < 10 个任务，每次 GET 约 50KB、POST 约 5KB，总耗时预估 3~10 秒。**用户能接受**（1~2 周才提交一次）。

**retry_per_task 默认 2**：用户没说具体次数，按 v2.0 §4.5 的策略（共重试 2 次，即最多请求 3 次）。理由：内网，网络真挂可能性低；业务失败（如 ViewState 过期）重试不自愈，多次只浪费时间。

### 9.4 Q1 修订：任务状态过滤位置确认

**v2.0 假设**：`tr[status]` 属性上有 status 字段，过滤规则待 Q1 答复。
**v2.1 实证**：抓包文件 `tools/captured/a4j_myTask_refreshBody.xml` 里能看到 `<tr class="body-row" id="..." status="RN" ...>`，**status 字段确认在 tr 标签的 `status` 属性上**。

**用户答复**：`status ∈ {未启动, 进行中}` 可填。

**架构变更**（最小）：

```python
# rdm/parsers.py
FILLABLE_STATUSES = {'RN', 'NS'}   # RN=进行中, NS=未启动

def is_task_fillable(task: dict, allow: set[str] = FILLABLE_STATUSES) -> bool:
    return task.get('status') in allow
```

**风险**：实际抓到的样本只有 `RN` 一种状态，`NS / FN / SP / FS` 等状态码是从 `tools/myTask.html` 的菜单按钮 ID 反推（`MENU_SU`=暂停、`MENU_FS`=强制终止 等），**真实 status 字段取值未在数据样本里全部验证过**。T-V2-003 实施时若遇到未知 status，应 logging warning 并默认**保留**该任务（宁滥勿缺，由验收时人工调整白名单），同时把样本贡献给团队。

### 9.5 Q7 修订：二次提交覆盖策略

**v2.0 假设**：Q7 未答，待定。
**v2.1 实证**：通过分析 `workLogView.js` 的 `synLogInfo()` 和 `saveLog()`——`_objectId` 字段填 reportId 时 RDM 走"更新"路径，自动覆盖现有日志数据；填空时新建。**Q7"覆盖"语义 RDM 原生支持，无需"先删再写"**。

**架构变更**：无（v2.0 的 `get_week_context()` 已经返回 `report_id`，submit 时透传即可）。

**已知行为说明**：
- 第一次 submit 时 reportId 为空，提交后 RDM 创建一条日志记录；
- 第二次 submit 时 GET workLogView 已经能拿到这条记录的 reportId；
- 但 §9.3 是**逐任务循环**，**第一个任务**提交后 reportId 创建出来；从**第二个任务**开始 GET 到的就是非空 reportId，自动走更新路径——逻辑天然衔接，无需特殊处理。

**未实现的能力（已知缺口）**：
- `_unplannedInfo` 编码格式里 `<taskId>{S3}` 单独出现表示"从本周日志里删除该任务行"（来自 `unplannedTask.js` 的 `isDel='Y'` 分支）。
- 当用户在前端表格里**删掉了某个任务**（之前填过工时，现在不想要了）时，仅提交剩余任务是**不会清掉旧数据**的——旧数据继续残留在 RDM。
- 本期**不暴露这个删除能力**——v2 需求里没提"删除某条已填工时"，超出范围。
- 留作未来扩展：若以后需要，在 `TaskEntry` 加 `delete: bool` 字段、`encode_unplanned_info` 检测到 `delete=True` 时输出 `<taskId>{S3}` 格式即可。

### 9.6 Q5、Q6 修订

- **Q5（任意日期自由选）**：API 层只需接受 `week_start` 为任意 YYYY-MM-DD（自动 normalize 到该周周一），v2.0 已满足。**前端**新增日期选择器（产品设计师范畴）。
- **Q6（记住用户名+密码）**：纯前端 localStorage，**不经过后端**。后端约束：不要在 cookie 里存明文密码（v2.0 决策 A 已经是加密形态，OK）。前端 localStorage 存密码 = 设备本地明文，安全风险**由用户接受**（本工具是个人提效工具，非多人共享场景）。建议产品设计师在登录页加复选框"记住密码（本地存储，仅本机生效）"和清除按钮。
